# Анализ конкурента: SecurOS Dispatcher v11

**Источник:** `Сompetitors/SecurOS_Dispatcher_User_Guide.pd_.pdf` (67 стр., User Guide), `Сompetitors/SecurOS-Dispatcher-Administration-Guide 3.pdf` (47 стр., Administration Guide). Сборка 133 от 24.05.2025. Производитель — Intelligent Security Systems (ISS).

**Назначение:** программная платформа для централизованного контроля и автоматизированного управления инцидентами (тревожными событиями) на крупных инфраструктурных объектах.

**Архитектура:** Web-клиент (браузерный, доступен с любого ПК сети) поверх серверной части, встроенной в платформу видеонаблюдения SecurOS. Видео и карта подаются через `WebConnect`. БД — **PostgreSQL** (только). Связь с клиентом — REST API.

---

## 1. Иерархия логических сущностей

Зоны контроля разнесены по 4 уровням:

| Уровень | Сущность | Что группирует |
|---|---|---|
| 1 | **Датчик** (sensor/детектор) | Источник событий (камера видеоаналитики, СКУД, ОПС и т.п.) |
| 2 | **Локация** | Группа датчиков по территориальному/иному признаку. Имеет адрес и номер. Может делиться на **Секторы**. |
| 3 | **Строение** | Группа локаций |
| 4 | **Пункт Управления (ПУ)** | Группа строений. К ПУ привязываются операторы. |

**Ключ:** оператор закрепляется за **одним или несколькими ПУ** и получает все инциденты со всех датчиков этих ПУ.

> **Этого нет в IM ITV** — там источник просто `objtype:objid` из Intellect, без формальной 4-уровневой иерархии.

## 2. Роли пользователей (5 ролей)

Многоуровневая фиксированная модель ролей:

### 2.1. Оператор
- Принимает и обрабатывает инциденты согласно регламенту в рамках своего ПУ.
- Может: принять, передать вышестоящему, отклонить.

### 2.2. Оператор Диспетчерского Центра (ОДЦ)
- Всё что Оператор +
- Просмотр Плановых работ.
- Просмотр Справочника (компании, сотрудники).
- Видит **историю действий** в карточке инцидента.

### 2.3. Старший Оператор Диспетчерского Центра (СОДЦ)
- Всё что ОДЦ +
- **Панель операторов** (видит всех, статусы — онлайн / оффлайн / на перерыве / запросы / просрочено).
- **Обработка запросов на перерыв** (одобрить/отклонить).
- **Передача инцидента оператору с конкретной ролью**.
- **Управление Плановыми работами** (создание, редактирование).
- **Управление Справочником** компаний и сотрудников.
- **Управление Локациями** (изменение уровней тревоги, отключение датчиков).
- **Архив инцидентов** + восстановление + массовое отклонение.
- **Просмотр истории действий операторов (Аудит)**.
- **Дашборд** со статистикой.

### 2.4. Оператор ГБР (группа быстрого реагирования)
- Получает инцидент только если его передали для выезда.
- Видит карту с местом инцидента и видео.
- Подтверждает или опровергает инцидент. Отправляет отчёт.
- Использует планшетные устройства (поэтому видео — через `WebConnect`, не напрямую).

### 2.5. Администратор
- Полный конфиг через отдельную панель администрирования (`http://<IP>:<port>/admin`).
- Импорт пользователей и датчиков из SecurOS.
- Создание типов инцидентов и сценариев.
- Настройка справочников, ролей, прав.

> **Этого нет в IM ITV** — там нет фиксированной ролевой модели, есть только матрица булевых прав на пользователя (`work_right`, `close_right`, `escalate_right` и т.д.) — гибче, но требует ручной настройки.

## 3. Жизненный цикл инцидента

### 3.1. Статусы (по админ-гайду §5.3)
- **Новый** — ожидает обработки.
- **Восстановленный из архива** — был в архиве, но восстановлен СОДЦ.
- **Переданный** — передан оператору на рассмотрение.
- **В работе** — оператор взял в обработку.
- **Архивный — Подтверждён** — закрыт по регламенту.
- **Архивный — Ложное срабатывание** — закрыт оператором как ложный.
- **Архивный — Штатная ситуация** — обоснованный, но без обработки по регламенту.
- **Архивный — Просрочен** — не обработан вовремя.

### 3.2. Авто-эскалация по таймауту
- При превышении **`Время ожидания принятия в работу`** (в минутах) инцидент **автоматически уходит вышестоящему**.
- Цепочка эскалации: **Оператор → ОДЦ → СОДЦ → Архив (Просрочен)**.
- Для ГБР цепочка короче: **ГБР → СОДЦ → Архив**.
- Время для СОДЦ настраивается отдельно (`Время ожидания для Старшего`).

### 3.3. Карточка инцидента содержит
- ID, название, дата/время возникновения, индикатор приоритета (низкий/средний/высокий).
- Информация о местоположении (Локация, Строение, адрес).
- Информация о датчике (тип, ID, название).
- Кнопки видео: «Смотреть архив (все)», «Смотреть онлайн (все)», «Местоположение на карте».
- Блок **Действия** — пошаговый сценарий обработки + подсказка с регламентом.
- Блок **Вложения** — файлы оператора (jpg, png, pdf, txt, doc, docx; ≤ 50 МБ).
- Блок **Комментарии** — переписка операторов.
- **История действий** (для ОДЦ+).
- Кнопки **Принять / Передать / Отклонить**.

### 3.4. Группировка инцидентов
- Все инциденты в ленте **сгруппированы по Локации** (по умолчанию).
- На панели группы — номер и название Локации, иконка плановых работ (если идут), цвет = максимальный приоритет в группе.
- Можно реагировать на **всю группу** или **только на один** инцидент.
- Лента поделена на 2 категории: **Не рассмотрено** / **В процессе**.

### 3.5. Закрытие инцидента
- **Принять → выполнить шаги регламента → Закрыть инцидент → указать причину закрытия → отправить в архив.**
- **Отклонить:** выбрать статус (Ложное срабатывание / Штатная ситуация) → выбрать причину из настроенного админом списка → пояснение (опционально или обязательно — настраивается).

### 3.6. Составной инцидент
- Тип инцидента может требовать **выполнения нескольких условий** (несколько событий с разных датчиков) в течение настраиваемого таймаута. До 4 дополнительных условий (5 всего).
- В таблице типов инцидентов коды событий разделяются `&`.

## 4. Сценарии обработки (Workflow)

### 4.1. Конструктор сценариев — главная фича
**Визуальный графический редактор блок-схем** (admin-панель, раздел «Создание сценария обработки»).

**Состав:**
- Заранее созданные **Действия оператора** двух типов:
  - **Действие** — простой шаг, требует подтверждения.
  - **Условие** — ветвление по выбору оператора Да/Нет.
- Действия располагаются на полотне drag-and-drop.
- Связи между блоками рисуются мышью (тянется линия от одного блока к другому).
- **Цепочка обязана начинаться** с блока **«Подтверждение инцидента»** и **заканчиваться** блоком **«Закрытие инцидента»**.

### 4.2. Регламент работ
- При создании типа инцидента указывается **текстовый регламент** (порядок действий, любая полезная информация).
- Поддерживает **HTML-теги `UL` и `LI`** для маркированных списков.
- Отображается оператору **в виде подсказки** в блоке Действия после взятия инцидента в работу.

### 4.3. Привязка
- Каждый тип инцидента → один сценарий.
- Один тип инцидента может назначаться **конкретному оператору** или **роли**.

> **Что слабее, чем у нас:** нет JSONLogic-условий — только бинарное Да/Нет. Сложные ветвления невозможны. У IM ITV это сделано в `CtrlLogic` через параметры `Msg` и `goto.{i}` / `macro.{i}` — гибче, но не визуализировано.

## 5. Уведомления и звуковые сигналы

- **Панель уведомлений** в навигационной панели, разделена на категории:
  - **Все** — общая лента.
  - **Критичные** — повышение уровня тревоги Локации до жёлтого/красного.
  - **Некритичные** — запросы на перерыв, отключение датчиков.
- Потерявшие актуальность уведомления исчезают автоматически.
- **Звуковой сигнал при получении инцидента** — настраивается админом (вкл/выкл).
- **Звуковой сигнал при получении уведомления** — настраивается отдельно.
- Уведомления о просроченных перерывах СОДЦ получает в виде специального сообщения.

> **Этого нет в IM ITV** — звуковых сигналов в коде не найдено, индикация только визуальная. У нас в WEB — пока тоже отсутствует.

## 6. Система перерывов операторов (уникальная фича!)

Целостная подсистема:

1. **Оператор нажимает «Перерыв»** → форма (промежуток времени + комментарий) → отправка.
2. **СОДЦ получает уведомление** → видит таймер «обработать», блок запроса с временем и комментарием.
3. **Решение СОДЦ:**
   - **Подтвердить** → инциденты переходят к другим операторам, у заявителя в интерфейсе таймер «до окончания перерыва».
   - **Отклонить** → оператор может перезапросить только через настраиваемый интервал.
4. **Если СОДЦ не решил вовремя** — авто-одобрение или авто-отклонение (настраивается админом).
5. **Если оператор не вернулся** к моменту окончания таймера — все вышестоящие операторы получают уведомление.
6. **Кнопка «Вернуться к работе»** — оператор подтверждает выход с перерыва.
7. **История перерывов** фиксируется в Аудите.

**Настраивается:**
- Время на принятие решения по запросу.
- Поведение при истечении времени (одобрить / отклонить).
- Время до возможности повторного запроса после отказа.

> **Этого нет ни у IM ITV, ни у нас.** Сильная организационная фича.

## 7. Плановые работы (Maintenance Windows)

**Назначение:** информационные заявки о ремонтных/обслуживающих мероприятиях на контролируемых Локациях.

**Состав заявки:**
- Период (начало–окончание).
- Локация(и) + Секторы.
- Тип работ (из настраиваемого справочника).
- Обслуживающая компания (из Справочника).
- Описание оборудования (опционально).
- Контакты (опционально).
- **Настройка получения инцидентов** во время работ — можно отключить датчики.

**Работа с заявками:**
- Создаёт СОДЦ (раздел «Плановые работы» → «Создать заявку»).
- Редактирует СОДЦ (предупреждение: изменения настроек вступают в силу немедленно).
- Просматривают ОДЦ и СОДЦ.
- Фильтры: по компании, дате, типу, локации, статусу (В процессе / Предстоящие / Прошедшие).
- В ленте инцидентов на Локации с активной плановой работой отображается специальная иконка.

> **Этого нет ни у IM ITV, ни у нас.** Решает реальную операционную проблему — снижение нагрузки во время ТО.

## 8. Справочник компаний и сотрудников

Двухуровневый справочник:

### 8.1. Раздел «Компании»
- Поля: Название, Описание, Телефон, E-mail, Руководитель, кол-во сотрудников, Дата добавления.
- CRUD: добавление, редактирование, фильтр по названию и периоду.

### 8.2. Раздел «Сотрудники компаний»
- Поля: ФИО, Описание (например, должность), Компания, Телефон, E-mail, Дата добавления.
- CRUD: фильтр по имени, компании, периоду.

**Управление:** СОДЦ. Просмотр: ОДЦ, ГБР.

**Использование:** связь компаний с плановыми работами (кто проводит ТО).

> **Этого нет ни у IM ITV, ни у нас.**

## 9. Управление Локациями (статус тревоги + отключение датчиков)

### 9.1. Уровни тревоги Локации (3 уровня)
- **Уровень 1 (синий)** — базовый.
- **Уровень 2 (жёлтый)** — высокий.
- **Уровень 3 (красный)** — критический.

При повышении уровня **все ответственные за Локацию операторы получают уведомление** под навигационной панелью; СОДЦ — кнопку «Перейти к Локации».

> Это **не приоритет инцидента**, а **состояние объекта** (например, при ЧС). Приоритет инцидента не зависит от уровня тревоги.

### 9.2. Включение/отключение датчиков
- СОДЦ может временно отключить получение событий с конкретных датчиков (точечно или массово через фильтр).
- При отключении инциденты с этих датчиков не формируются.
- Все ответственные операторы уведомляются.
- Используется для снижения нагрузки во время плановых работ.

> **Этого нет ни у IM ITV, ни у нас.** Дополнительное измерение управления потоком инцидентов.

## 10. Архив инцидентов

### 10.1. Поиск (фильтры)
- Дата и время.
- Приоритет.
- Датчики (типы).
- Роль оператора.
- Конкретный оператор.
- Тип инцидента.
- Статус (Подтверждён / Ложное / Штатная / Просрочен).
- Пункт Управления.
- Локация.
- Чекбокс «Плановые работы».

### 10.2. Шаблоны фильтров
- Текущий набор фильтров можно сохранить как именованный шаблон.
- Шаблон выбирается из выпадающего списка → фильтры применяются автоматически.
- Шаблон **нельзя редактировать**, только удалить и создать новый.

### 10.3. Карточка архивного инцидента
- Полная история действий.
- Отчёты, вложения.
- Кнопка **«Вернуть в работу»** (восстановление одиночное или массовое).
- Восстановленные инциденты направляются доступному оператору (выбрать конкретного нельзя).

### 10.4. Массовые операции
- **Массовое восстановление** инцидентов (отметить флажки → «Вернуть в работу»).
- **Массовое отклонение просроченных** — указать статус, причину, пояснение.

### 10.5. Отчёты
- **По одному инциденту** — кнопка «Скачать PDF».
- **По группе инцидентов** — отметить флажками → «Полный отчёт» / «Краткий отчёт».
- При отключённом параметре «Экспорт отчётов в PDF» данные направляются в SecurOS, где обрабатываются скриптами Node.JS (кастомные отчёты в сетевую папку).

> **У IM ITV** — отчёты через RDLC (более гибкие шаблоны), но без шаблонов фильтров и массовых операций. **У нас** — пока только action `generateReport` в DSL без UI.

## 11. Дашборд / Статистика инцидентов

Вкладка **Дашборд** в разделе Архив. 4 диаграммы по выбранным фильтрам:

1. **Общее количество инцидентов** — круговая по статусам.
2. **Количество инцидентов по Локациям** — гистограмма или кластеры.
3. **Количество инцидентов по типу** — гистограмма или кластеры.
4. **Количество инцидентов ПУ по операторам** — гистограмма + **среднее время реакции операторов** на инциденты (по каждому ПУ).

> **Этого нет ни у IM ITV (RDLC ≠ дашборд), ни у нас.** Простой, но важный BI-инструмент для руководителя.

## 12. Аудит (история действий операторов)

Раздел **Аудит** для СОДЦ и Администратора.

**Типы фиксируемых событий:**
- Смены (вход/выход оператора, начало/конец перерыва, запрос/решение о перерыве).
- Создание плановой работы.
- Редактирование плановой работы.
- Включение датчика Локации.
- Выключение датчика Локации.
- Изменение уровня тревоги Локации.
- Действие с инцидентом (со ссылкой на карточку).

**Фильтры:** дата/время, тип события, роль, оператор, Локация.

**Колонки таблицы:** Дата, Время, Тип события, Оператор (имя+роль), **Детали события** (для действий с инцидентом — название, переход статуса, действие).

**Доступность вкладки** — настраивается админом.

> **У IM ITV** отдельного модуля аудита нет, история раскидана по полю `Works`/`PROTOCOL_INC_SERVER`. **У нас** — поле `history` в `scenarioResult`, но без UI.

## 13. Передача инцидента оператору с конкретной ролью

Только для СОДЦ. В карточке инцидента → «Передать» → выпадающий список ролей → выбрать роль → передача.

**Особенность:** в списке отображаются **только операторы, прикреплённые к ПУ инцидента**.

## 14. Видео и карта (через WebConnect)

### 14.1. Источники видео в карточке инцидента
- **«Смотреть архив (все)»** — открывает архив всех привязанных к датчику камер на момент возникновения инцидента в **Медиа Клиенте**.
- **«Смотреть онлайн (все)»** — живое видео в Медиа Клиенте.
- **«Местоположение на карте»** — переход к датчику в окне «Карта: Интерфейс оператора».

### 14.2. ГБР
- Использует встроенный **`WebConnect`** (HTML5-видео и карта прямо в браузере, без внешнего клиента).
- В карточке: блок «Локация» с кнопкой **Карта** (выбор уровня) и кнопкой **Видео** (с переключением между связанными камерами).
- Подходит для планшетов.

### 14.3. Привязка камер к датчику
- Делается админом в разделе «Датчики» при импорте/редактировании.
- Для каждого датчика — список **«Связанные камеры»**.
- Просмотр видео доступен оператору по этим камерам в карточке инцидента.

## 17. Архитектурная привязка к SecurOS VMS — ключевая стратегическая слабость

SecurOS Dispatcher — это **не самостоятельный продукт**, а **Модуль внутри платформы SecurOS VMS**. Без полноценной инсталляции SecurOS Видеосервера он не запускается. Это разворачивается на двух уровнях.

### 17.1. На уровне источников (датчики, камеры, пользователи) — жёсткая привязка к SecurOS

| Сущность Dispatcher | Откуда берётся / на что опирается |
|---|---|
| **Датчики** | `Импортировать датчики` → импорт объектов из конфигурации SecurOS (§5.2.8) |
| **Типы датчиков** | Импорт из **`ddi-файлов`** SecurOS (§5.2.9) |
| **Типы событий** | Идентификатор `Код типа события` = код события объекта SecurOS (§5.2.10) |
| **Камеры в карточке инцидента** | `Связанные камеры` ссылаются на SecurOS-объект `Камера` (§4.1.7) |
| **Карта** | Объект `Карта` SecurOS (§4.1.4), не GIS |
| **Пользователи (Персонал)** | Импорт из учёток SecurOS; `ID Прав пользователя`, `ID Отдела` ссылаются на объекты `Дерева объектов` SecurOS (§5.2.7) |
| **Видео в браузере** | `WebConnect` SecurOS (§4.1.3) |
| **Долговременный архив видео** | Объект `Архиватор` SecurOS (§4.1.7) |
| **БД** | Только PostgreSQL, схема создаётся объектом `База данных` SecurOS (§4.1.1) |
| **Связь с ядром** | IIDK / REST API → `Dispatcher: Сервер`, исполняющийся внутри Видеосервера SecurOS |

**Список обязательных объектов SecurOS, без которых модуль не запустится:**
`База данных`, `REST API`, `WebConnect`, `Карта`, `Image Processor`, `Dispatcher: Сервер`; на рабочем месте оператора — `Медиа Клиент`, `Карта: Интерфейс оператора`, `Dispatcher: Интерфейс оператора`.

### 17.2. На уровне операционной иерархии (Локация / Строение / ПУ) — отдельный слой

В этом отношении архитектура SecurOS Dispatcher **сделана правильно**: операционная иерархия `Локация → Строение → Пункт Управления` живёт **внутри Dispatcher**, а не в SecurOS. Админ создаёт её вручную в админ-панели и привязывает к ней импортированные датчики через mapping-таблицу.

То есть VMS-топология (где физически камера в дереве SecurOS) ≠ операционная топология (за какое здание/ПУ отвечает диспетчер). Это правильный паттерн **разделения VMS-слоя и operational-слоя**, который мы должны унаследовать.

### 17.3. Что это значит на практике

- **Не работает поверх Axxon Next, Axxon One, Milestone, Genetec, ONVIF** — только SecurOS.
- **Не работает в гетерогенной инфраструктуре** клиента (где один объект на одной VMS, другой на другой).
- **Авторизация только через SecurOS Users** — нет AD/LDAP/SSO.
- **Карта** — закрытый объект SecurOS, не интегрируется с публичными GIS.
- **Видеоплеер** — только через `WebConnect` SecurOS.

**Это ключевая стратегическая слабость SecurOS Dispatcher на тендерах**, где у клиента уже стоит другая VMS, нужна гетерогенная инфраструктура, или есть требование SSO / open standards / multi-vendor стратегии.

### 17.4. Наша архитектурная позиция (WEB + Go-бэкенд)

| Слой | SecurOS Dispatcher | Наш WEB + Go |
|---|---|---|
| **Источники событий** | Жёстко привязаны к SecurOS VMS | **VMS-агностичный adapter-слой** в Go-бэкенде: Axxon One, Axxon Next, SecurOS, Milestone, Genetec, ONVIF, СКУД, ОПС, кастомные REST/MQTT/Kafka |
| **Operational hierarchy** | Свой слой Dispatcher (правильно) | **Тот же подход** — собственный operational layer, не зависит от того, какая VMS под капотом |
| **Идентификация пользователей** | Только SecurOS Users | AD/LDAP/SAML/OAuth2/OIDC + локальные + импорт из любой VMS |
| **Карта** | Растровая `Карта` SecurOS | GIS (Leaflet/Mapbox/OpenLayers/Yandex/Google), независимо от VMS |
| **Видео-плеер** | Только WebConnect SecurOS | Свой WebRTC/HLS-плеер с адаптерами на любые VMS-источники |
| **БД** | Только PostgreSQL | PostgreSQL по умолчанию + потенциально совместимые СУБД через миграции |

**Стратегическая формула:** «Менеджер инцидентов AxxonSoft работает поверх любой VMS — не заставляет менять то, что у вас уже есть». Эта позиция полностью закрывает SecurOS Dispatcher на тендерах с гетерогенной инфраструктурой.

Подробное предложение по реализации — `web-feature-recommendations.md` § P2.11 «VMS-агностичные адаптеры источников».

## 18. Что вызывает вопросы / слабые стороны SecurOS Dispatcher

| # | Слабость | Почему критично |
|---|---|---|
| 1 | **Сценарии — только Да/Нет ветвления.** Нет сложных условий типа «если приоритет=High AND время=ночь». | Жёстко ограничивает гибкость. |

| 5 | **Только PostgreSQL.** Жёсткая привязка СУБД. | Ограничение для энтерпрайза с другими стандартами. |

| 10 | **Дашборд — только встроенный**, без экспорта данных или интеграции с BI. | Не масштабируется. |
| 11 | **Нет мультитенантности** в явном виде в гайдах. | Ограничение для облачного развёртывания. |
| 12 | **Работает только поверх SecurOS VMS** (см. §17). Не запускается на Axxon Next, Axxon One, Milestone, Genetec, ONVIF. | **Главный проигрыш на тендерах с другой/гетерогенной VMS.** |

| 14 | **Карта** — это объект SecurOS (`Карта: Интерфейс оператора`), не GIS, не OSM, не Яндекс/Google. | Закрытая экосистема. |
| 15 | **Версионирование сценариев — не описано в гайдах.** Очевидно, что правки регламента «вступают в силу немедленно». | Невозможно безопасно править рабочие сценарии. |

## 19. Что у SecurOS Dispatcher выдающегося (TOP-10 «фишек»)

Сильные стороны, которые делают его прямым серьёзным конкурентом:

1. **Полная иерархия сущностей** Датчик → Локация → Строение → ПУ — сразу даёт правильный домен для крупных объектов (стадион, завод, аэропорт).
2. **Чёткая ролевая модель из 5 ролей** с понятным распределением полномочий — операционно зрелая.
3. **Авто-эскалация по таймауту** между ролями — встроена и настраиваема.
4. **Подсистема перерывов операторов** с одобрением, авто-решением, контролем возвращения, уведомлениями просрочки.
5. **Плановые работы** с привязкой к компаниям и временным отключением датчиков.
6. **Справочник обслуживающих компаний и сотрудников** — связан с плановыми работами.
7. **Уровни тревоги Локаций** (синий/жёлтый/красный) — отдельное измерение от приоритета инцидента.
8. **Дашборд со статистикой по Локациям, типам, операторам и средним временем реакции.**
9. **Полный аудит действий** (вход/выход, перерывы, инциденты, плановые работы, изменения уровня тревоги, отключения датчиков) — отдельная вкладка с фильтрами.
10. **Шаблоны фильтров для архива** — экономия времени при типовых отчётных запросах.

**Бонусы:**
- Темы (светлая/тёмная), смена языка, **ГБР через WebConnect** для планшетов.
- Составные инциденты (несколько условий с разных датчиков в окне таймаута).
- Массовые операции в архиве (восстановить, отклонить).
- Звуковые сигналы для инцидентов и уведомлений (отдельные настройки).

## 20. Резюме для нашей команды

SecurOS Dispatcher — это **зрелый продукт с богатым операционным функционалом**, ориентированный на крупные диспетчерские центры с многоуровневой иерархией ответственности. Сильные стороны — **организационная зрелость** (роли, перерывы, плановые работы, аудит) и **полнота справочников и сущностей**.

**Слабые стороны** — **жёсткая привязка к SecurOS VMS** (§17), низкая гибкость сценариев (только Да/Нет), отсутствие SSO, отсутствие версионирования сценариев, отсутствие открытого API для конфигурации.

**Стратегия для нас в WEB + Go-бэкенд:**
- **Перенять** все 10 «фишек» из §19.
- **Превзойти** в гибкости сценариев (JSONLogic), версионировании, открытости (API/CLI/Git-friendly формат), мультитенантности и SSO/RBAC.
- **Стать VMS-агностичными** (см. §17.4) — единственное решение на рынке, работающее поверх любой VMS (Axxon, SecurOS, Milestone, Genetec, ONVIF и др.). Это убирает SecurOS Dispatcher с тендеров с гетерогенной инфраструктурой.
- **Оставить** наш сильный задел — Server-Driven UI, JSON-сценарии, мультиплатформенный runner (Web/iOS/Android), редактор с undo/redo и live preview.

Детальное сопоставление — см. `competitor-comparison.md`.
Список конкретных фич к реализации — см. `web-feature-recommendations.md`.
