# Демонстрационные сценарии DSL v1

> 10 операторских сценариев + 3 библиотечных под-сценария + папка [`architecture/`](./architecture/) с 15 архитектурными артефактами.
> Бизнес-домен: **VMS / системы безопасности** — камеры, пожарная сигнализация, СКУД.
> Все сценарии **валидируются** по [`../dsl-v1-schema.json`](../dsl-v1-schema.json).
>
> Если нужно показать **архитектурные свойства** DSL (inline-резолв sub-scenarios, версионирование, анти-паттерны, читаемость diff'а) —
> см. отдельную папку [`architecture/`](./architecture/) и её [README](./architecture/README.md).
>
> Используются:
> - для защиты архитектурных решений с продактом;
> - как тестовые данные для будущего операторского Web-приложения и редактора;
> - как «smoke-тесты» при изменениях DSL — если правишь схему, прогоняй
>   валидацию всех 13 файлов (см. ниже команду).

## Как запустить валидацию

Из корня пакета (`incident-manager-dsl/`):

```powershell
npm install --silent ajv ajv-cli ajv-formats
Get-ChildItem -Path examples -Filter "*.json" | ForEach-Object {
  npx ajv validate --spec=draft2020 -c ajv-formats -s dsl-v1-schema.json -d $_.FullName --strict=false
}
```

Ожидаемый результат: **13 строк `... valid`.**

## Покрытые ключевые архитектурные решения DSL v1

Каждое решение из [`../dsl-v1-draft.md`](../dsl-v1-draft.md) §13 продемонстрировано в одном или нескольких сценариях:

| Решение | Где видно в примерах |
| --- | --- |
| **Р1.** `scenarioResult` целиком в jsonb (без отдельной таблицы событий) | runtime-семантика; в файлах сценариев напрямую не отражено |
| **Р2.** События actions пишутся в `history` runner'ом | runtime-семантика; в файлах сценариев напрямую не отражено |
| **Р3.** `dslVersion` только внутри script | поле `dslVersion: "1.0"` в каждом файле |
| **Р4.** Sub-scenario: гибрид + pin к версии | 05 (один child), 06/10 (несколько children); все ссылки с `version: 1` |
| **Р5.** Запрет циклов CallScenario | 13 файлов — циклов нет (DFS-обход на сервере подтвердит) |

## Покрытые принципы

| Принцип | Видно в |
| --- | --- |
| **П1.** Платформо-независимость | везде (нет CSS/px/нативных контролов) |
| **П2.** Только семантические подсказки в view | везде (label, required, options) |
| **П3.** Все эффекты через Action | 01–10 (никаких `is_finish` флагов) |
| **П4.** Правило трёх рендереров | везде (каждое поле имеет очевидный рендер на Web/iOS/Android) |
| **П5.** Стабильные id опций | везде (`option.id` строки, не индексы) |
| **П6.** Декларативный граф | везде (JSONLogic-выражения вместо кода) |
| **П7.** Сервер не знает о DSL | везде (script — непрозрачный jsonb) |
| **П8.** Forward-compat через игнор | (продемонстрировать в runtime-тестах) |
| **П9.** Sub-scenario резолвится сервером | 05, 06, 10 |

## Что НЕ покрыто (намеренно)

Этих фич нет в DSL v1 — соответственно, в примерах их тоже нет:

- **Lifecycle-actions** (`onEnter`/`onExit`) — отложены в v2.
- **Validation rules внутри шага** (regex для Comment, и т.п.) — v2.
- **Параллельные ветки (BPMN fork/join)** — v2.
- **Custom JSONLogic operators** — v2.
- **`labelKey` для i18n** — v2.
- **`inputs`/`outputs` для CallScenario** — v2.
- **Latest-ссылки в CallScenario** (без version) — v2.

## Что нужно дополнительно протестировать на этих примерах

Когда будет готов **редактор сценариев**:
- открыть каждый файл — видны ли все шаги корректно;
- попробовать удалить опцию — увидеть, что условные переходы помечены сломанными;
- попробовать создать цикл (вставить #06 как sub-scenario в lib_01) — должна быть ошибка валидации.

Когда будет готов **runner Web** (операторское приложение):
- пройти каждый сценарий end-to-end, проверить, что `scenarioResult.history`
  содержит все ожидаемые события;
- убедиться, что после inline-резолва на сервере (#05, #06, #10) runner
  видит плоский набор шагов без `CallScenario`;
- проверить, что условные переходы работают корректно для всех 5 путей в #09.

Когда будет готов **сервер**:
- сохранить все 13 сценариев — все должны пройти валидацию циклов
  (см. [`../proposal-versioning.md`](../proposal-versioning.md) §6.2);
- создать инцидент по #05/#06/#10 — проверить, что в `incidents.scenario`
  записан плоский граф с префиксированными id (e.g. `tech_support__select_equipment`);
- проверить, что при правке `lib_01` версия родителей **не** меняется автоматически — pin удерживает.
