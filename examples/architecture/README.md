# Архитектурные примеры DSL v1

Эта папка — **дополнение** к `examples/01..10` (продакшн-сценарии VMS). Если те
показывают **выразительность** DSL на реальных задачах, то эти — **архитектурные
свойства**: что именно DSL обещает, как он защищается от ошибок и где границы.

Аудитория: тех-лиды, ревью продуктовой архитектуры, разработчики runner'ов и
сервера.

## Карта артефактов

| Артефакт | Что доказывает | Опирается на пункт DSL-спеки |
| --- | --- | --- |
| `A1-minimal-valid-scenario.json` | DSL не заставляет писать boilerplate. Минимальный сценарий = 1 шаг + 1 finish. | Вся спека читается на этом примере как «понятная по умолчанию». |
| `A5-anti-patterns/` | Граница между «ловит схема», «ловит сервер семантически» и «не ловит ничто, нужен code review». | §11 «Уровни валидации», открытый вопрос про fwd-compat |
| `A7-diff-readability/` (2 файла) | Декларативность даёт читаемый diff между версиями сценария. | Преимущество SDUI-подхода (см. `market-research.md`) |

## Как читать

### A1: «насколько просто»

Открыть `A1-minimal-valid-scenario.json`. **Минимальный валидный сценарий —
несколько строк JSON.** Это нижняя граница. Любой более сложный сценарий —
это явный выбор автора, а не требование схемы.

### A5: «что DSL умеет ловить, а что — нет»

Это самая ценная часть для ревью продакта/архитектора. Не каждая ошибка ловится
на одном уровне — в спеке есть **три уровня валидации**:

| Уровень | Реализован | Что ловит |
| --- | --- | --- |
| 1. JSON Schema | `dsl-v1-schema.json` | Структуру, типы, обязательные поля, whitelist'ы констант |
| 2. Семантическая | сервер при `POST /scenarios`, редактор при сохранении | Висящие `goto`, дубликаты `step.id`, недостижимые шаги, тупиковый default |
| 3. Стилистическая | code review + редактор-linter (отложено) | «Сценарий валидный, но плохо спроектирован» |

Файлы и их матчинг с уровнями:

| Файл | Какой уровень должен поймать | Реальное поведение |
| --- | --- | --- |
| `bad-1-schema-broken-goto.json` | Уровень 2 (семантика) | Схему проходит. Сервер должен ответить 422. |
| `bad-3-unknown-action.json` | Уровень 1 (JSON Schema) | **Отвергается схемой** (validator fails). |
| `bad-4-numeric-option-ids.json` | Уровень 3 (стиль) | Схему проходит, сервер пропустит. Нужен design-time linter в редакторе. |
| `bad-5-deeply-nested-rules.json` | Уровень 3 (стиль) | Схему проходит, сервер пропустит. Нужен code review и архитектурная рекомендация «разбивайте на шаги». |

### A7: «diff читаем»

Сделать `git diff --no-index A7-diff-readability/v1-original.json
A7-diff-readability/v2-after-edit.json` или открыть оба файла в IDE-diff. Вся
правка — 4 локальных места, каждое тривиально объясняет суть. Это аргумент в
пользу SDUI/декларативного подхода.

## Воспроизведение валидации

```bash
cd incident-manager-dsl
python -c "import jsonschema,json,glob
schema=json.load(open('dsl-v1-schema.json',encoding='utf-8'))
for f in sorted(glob.glob('examples/architecture/**/*.json',recursive=True)):
    try:
        jsonschema.validate(json.load(open(f,encoding='utf-8')),schema)
        print('OK   ',f)
    except jsonschema.ValidationError as e:
        print('FAIL ',f,'::',e.message[:80])"
```

Ожидаемый результат: все OK, кроме `bad-3-unknown-action.json` (FAIL по схеме —
ожидаемое поведение).

## Открытые архитектурные вопросы, обнаруженные при подготовке примеров

### Forward-compat для `Action`

Текущая схема использует строгий `Action.oneOf` с пятью whitelist'ed-типами.
Это значит: **ни одного нового action нельзя добавить runner'у первым** — новый
action отвергается схемой при `POST /scenarios`. Сначала нужно обновить схему,
потом сервер, потом все runner'ы.

Альтернатива (open discriminator) даёт настоящий forward-compat (runner
игнорирует неизвестное), но плата — опечатка в `type` не ловится схемой.

**Зафиксированное решение:** оставить строгий `oneOf`. Расширение Action-набора
= RFC + bump `dslVersion` + синхронизированный релиз.

### Шаг с одной опцией

Сейчас `RadioButton.view.options` имеет `minItems: 2`. Это разумно: одна опция
бессмысленна. Но в реальности встречаются «псевдо-опции» — кнопка
подтверждения, оформленная как radio. Кейс не критичен, но если продакт
покажет такие сценарии — нужно решать.

### Linter в редакторе

Анти-паттерны `bad-4` и `bad-5` показывают: **нужен design-time linter**, не
только runtime-валидация. Это отдельная задача редактора, не DSL. Зафиксировать
как требование к редактору.
