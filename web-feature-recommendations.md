## P0. Обязательный фундамент <a id="p0"></a>

Без этого WEB-продукт нельзя продавать на рынке диспетчерских центров.

### P0.1. Иерархия объектов: Датчик → Локация → Строение → Пункт Управления

**Источник:** SecurOS §1.
**Слой:** Сервер + Конфигуратор.

**Что нужно:**
- Модель данных на сервере: 4 уровня иерархии (sensors, locations, buildings, control_points), у Локации — поле «адрес», `sectors[]` и связь с Camera/`source` через mapping-таблицу.
- В DSL — расширить (или вынести наружу!) механизм связывания триггеров с этими сущностями. **Решение для обсуждения:** сама иерархия — на сервере, а DSL получает в `context` только активные значения (`location.id`, `building.id`, `sensorId` и т.п.) для использования в JSONLogic.
- Конфигуратор иерархии — отдельный раздел в admin-UI.

**Acceptance:** оператор видит инциденты только своих ПУ; в карточке отображаются Локация, Строение, адрес, Сектор; фильтр архива по уровням иерархии работает.

### P0.2. Ролевая модель + RBAC

**Источник:** SecurOS §2 + IM ITV per-PERSON флаги.
**Слой:** Сервер + Конфигуратор + Раннер.

**Что нужно:**
- 5 преднастроенных ролей **как пресеты**: Оператор / ОДЦ / СОДЦ / ГБР / Администратор.
- **Гибридная модель:** роли + кастомные пермишны (можно отключить отдельную возможность у роли для конкретного юзера).
- Привязка пользователя к ПУ.
- Отдельный admin-UI: «Персонал» (импорт из источника, редактирование, активация).
- API: `users`, `roles`, `permissions`, `pu_assignments`.
- Раннер: показывает только разделы и кнопки, доступные роли.

**Атомарные пермишны (минимум, по образцу IM ITV `DlgRightIM`):**
- `incident.work` — взять в работу.
- `incident.close` — закрыть.
- `incident.suspend` — приостановить.
- `incident.observe` — просматривать (read-only).
- `incident.assign` — назначить другому.
- `incident.escalate` — эскалировать.
- `incident.work_escalated` — работать с эскалированными.
- `incident.filter` — изменять фильтры/группировку у других.
- `incident.auto_unassign` — снимать назначение.
- `archive.delete` — массовое удаление инцидентов из архива (см. P0.8).
- `session.kick` — принудительно завершать сеансы других операторов.
- `config.modify_scenario` / `config.modify_users` / `config.modify_hierarchy` — конфигурационные права.

**Опции безопасности сеансов (по образцу SecurOS §16):**
- **Запрет одновременного входа** одного пользователя с разных устройств (вкл/выкл, опция админа).
- **Принудительное завершение сеанса требует ввод пароля** (опция админа, по умолчанию выкл — UX-приоритет).
- **Принудительный kick** оператора у СОДЦ (через `session.kick` пермишн); kick’нутый видит modal с обоснованием.

**Acceptance:** все 5 ролей из SecurOS работают; админ может «отозвать» одну привилегию у конкретного оператора без выдумывания новой роли; СОДЦ может принудительно завершить сеанс зависшего оператора.

### P0.3. Жизненный цикл инцидента + статусы + резолюции

**Источник:** SecurOS §3.1, IM ITV `Status` enum.
**Слой:** Сервер.

**Что нужно:**
- Статусы (минимум): `new`, `assigned`, `in_progress`, `transferred`, `closed_confirmed`, `closed_false`, `closed_normal`, `closed_overdue`, `restored_from_archive`.
- Состояния можно расширять админом (custom statuses) — конфигурируется на сервере (`incident_action_transitions`).
- API эндпоинты: `POST /incidents/:id/transition`, `GET /incidents/:id/history`.
- Привязать переходы статуса к действиям DSL: `finish` → автоматически выставляет `closed_*`; `escalate` → `assigned` другому юзеру/роли; `assign` → `transferred`.

**Причины отклонения и пояснения (по SecurOS §3.5, §15.3):**
- Справочник причин с двумя категориями: **«Ложное срабатывание»** и **«Штатная ситуация»**.
- Причины настраиваются админом (CRUD), могут быть включены/выключены, привязаны к конкретным типам инцидентов или общие.
- Глобальный параметр **«Обязательное наличие пояснения при отклонении»** (вкл/выкл).
- При закрытии инцидента — выбор причины + опциональное/обязательное текстовое пояснение.

**Визуальная индикация просрочки (по SecurOS §3.4 + IM ITV `TimeExpiredColor`):**
- В ленте инцидентов — **обратный отсчёт** времени до автоэскалации (P0.4).
- При истечении 80% времени — оранжевая подсветка строки.
- При истечении 100% — красная подсветка + автопередача (см. P0.4).
- В карточке инцидента — крупный таймер «Осталось до эскалации».

**Acceptance:** все 8 статусов SecurOS реализованы; история переходов доступна; оператор отклоняет инцидент через причину «Ложное срабатывание → Ошибка датчика» с обязательным пояснением; в ленте видно, что инцидент скоро просрочится.

### P0.4. Автоматическая эскалация по таймауту

**Источник:** SecurOS §3.2 + IM ITV `spRunAutoEscalate`.
**Слой:** Сервер + DSL.

**Что нужно:**
- DSL уже имеет `timers.escalateAfterSec` — оставить.
- Сервер: фоновая задача (cron каждую секунду как у IM ITV, либо event-driven через таблицу с дедлайнами).
- Цепочка эскалации: Оператор → ОДЦ → СОДЦ → Архив (Просрочен). Настраивается **в админ-панели для типа инцидента** (можно нестандартную цепочку).
- При попадании в Архив → статус `closed_overdue`, фиксация в аудите.

**Acceptance:** инцидент, который никто не взял в работу 5 минут, автоматически уходит ОДЦ; ещё 5 минут — СОДЦ; ещё 10 — в архив.

### P0.5. Группировка инцидентов в ленте + реакция на группу

**Источник:** SecurOS §3.4 + IM ITV `EventGroup`.
**Слой:** Раннер + Сервер.

**Что нужно:**
- Лента у Оператора и СОДЦ группируется (по Локации по умолчанию).
- Для группы — заголовок (номер Локации, **цвет фона/рамки заголовка = max приоритет инцидентов в группе**, иконка плановых работ при активной заявке P1.2).
- Кнопки на группе: Свернуть/Развернуть, Принять всю группу, Передать всю группу, Отклонить всю группу.
- 2 категории сверху: «Не рассмотрено» / «В процессе».
- **Опционально настраиваемый признак группировки** — выпадающий список в шапке ленты, минимальные значения (по образцу IM ITV `EventGroup.EGroup`):
  - **По Локации** (default — как SecurOS).
  - **По типу инцидента** (`BY_EVENT`).
  - **По источнику-объекту** (`BY_OBJECT`).
  - **По обработчику/назначенному оператору** (`BY_HANDLER` / `BY_OPERATOR` — есть в legacy IM ITV `DlgGrouping_IM`, отсутствует в SecurOS).
  - **По приоритету** (наш эксклюзив).
  - **Без группировки** (плоский список).

**Acceptance:** взятие группы в работу за один клик — все инциденты группы переходят в `in_progress` атомарно; при наличии в группе High-инцидента заголовок группы красный; при активной плановой работе — иконка возле номера Локации.

### P0.7. Прикрепление файлов + комментарии к инциденту

**Источник:** SecurOS §3.3.
**Слой:** Сервер + Раннер.

**Что нужно:**
- Загрузка файлов: jpg, png, pdf, txt, doc, docx, xlsx, csv, log (расширить список).

- **Эксклюзив:** **антивирус-проверка** через ClamAV / ICAP перед сохранением; DLP-теги; auto-detection PII.
- В карточке: блоки «Вложения» и «Комментарии» рядом, как у SecurOS.

**Acceptance:** оператор прикрепляет PDF-протокол; коллега видит, скачивает, оставляет комментарий с упоминанием.

### P0.8. Архив инцидентов с фильтрами

**Источник:** SecurOS §10 + IM ITV PROTOCOL.
**Слой:** Сервер + Конфигуратор/Раннер (роль СОДЦ).

**Что нужно:**
- Полный набор фильтров (см. SecurOS §10.1): дата, приоритет, датчики, роль, оператор, тип, статус, ПУ, Локация, чекбокс «во время плановых работ».
- Серверный поиск (PostgreSQL JSONB + GIN-индексы).
- Карточка архивного инцидента: вся история действий, отчёты, вложения, история комментариев.
- Кнопка «Вернуть в работу» (одиночная).
- **Бонус:** массовое восстановление и массовое отклонение (см. P0.9).

**Административные операции архива (SecurOS §15.6 + IM ITV `RESEND_EVENT`):**
- **Массовое удаление** инцидентов админом по периоду + фильтру статусов (требует `archive.delete` пермишн из P0.2). Подтверждение с показом количества подлежащих удалению. Действие фиксируется в Аудите (P1.6) с обоснованием. Удалённое **не восстанавливается** — предупреждение обязательно.
- **«Переотправить событие»** в контекстном меню архивного инцидента — пересоздать инцидент на основе исходного события (источник, тип, параметры). Полезно для отладки регламентов и ретеста сценариев. Признак переотправки фиксируется в новом инциденте (`reposted_from_id`).

**Acceptance:** СОДЦ находит «инциденты типа Х за последний месяц на ПУ-3, обработанные оператором Иванов» за несколько кликов; админ удаляет ~5000 тестовых инцидентов за период пусконаладочных работ; разработчик регламента «переотправляет» архивный инцидент после правки сценария для проверки.

## P1. Перенять у SecurOS все 10 фишек <a id="p1"></a>

Без этих 10 — мы хуже SecurOS на демо.

### P1.1. Подсистема перерывов операторов

**Источник:** SecurOS §6.
**Слой:** Сервер + Раннер.

**Что нужно:**
- Кнопка «Перерыв» в шапке у роли Оператор/ОДЦ.
- Форма запроса: период + комментарий.
- **СОДЦ** видит запросы в панели уведомлений (категория «Некритичные»), может одобрить/отклонить.
- Таймер «обработать» → авто-решение по настройке.
- При одобрении: оператор видит таймер «до окончания перерыва», его инциденты автопередаются другим (round-robin или least-loaded).
- Кнопка «Вернуться к работе» завершает перерыв.
- При просрочке возвращения — уведомление всем СОДЦ.
- Запрет повторного запроса после отказа на N минут.
- История перерывов в Аудите.

**Эксклюзив:** **планирование** перерывов на будущее (с автоматическим срабатыванием), **обед-расписание** (повторяющиеся ежедневные интервалы).

**Acceptance:** оператор уходит на обед, инциденты перераспределены автоматически, СОДЦ видит обзор «кто и когда на перерыве».

### P1.2. Плановые работы

**Источник:** SecurOS §7.
**Слой:** Сервер + Конфигуратор/Раннер.

**Что нужно:**
- Раздел «Плановые работы» (для ОДЦ+).
- Заявка: период, Локация(и) + Сектор(ы), Тип работ, Компания, Оборудование, Контакты, **галочка временного отключения датчиков**.
- Фильтры: компания, дата, тип, локация, статус (В процессе/Предстоящие/Прошедшие).
- Иконка плановых работ в заголовке группы инцидентов.
- При активной работе — опциональное **скрытие или подсветка** инцидентов с этих датчиков.
- В архиве — фильтр «во время плановых работ».

**Эксклюзив:** **уведомление за N минут до начала** (по e-mail/Telegram/SMS); чек-лист завершения работ; **автозакрытие** с генерацией отчёта.

**Acceptance:** СОДЦ создаёт заявку на ТО на воскресенье 02:00–06:00, в это время датчики отключены, в архиве инцидентов нет ложных срабатываний.

### P1.3. Справочник обслуживающих компаний и сотрудников

**Источник:** SecurOS §8.
**Слой:** Сервер + Конфигуратор.

**Что нужно:**
- Сущности: `companies` (название, описание, телефон, e-mail, руководитель, дата добавления) и `company_employees` (ФИО, описание, компания, телефон, e-mail).
- CRUD для СОДЦ (полный) и ГБР/ОДЦ (просмотр).
- Связь с Плановыми работами и опционально с Камерами/Локациями (ответственный за оборудование).

**Эксклюзив:** **импорт компаний/сотрудников из CSV/Excel/CRM**, синхронизация с 1С/SAP, **фотография сотрудника**.

**Acceptance:** при создании плановой работы выбирается компания из справочника, контакты подставляются автоматически.

---

### P1.4. Уровни тревоги Локаций + управление датчиками

**Источник:** SecurOS §9.
**Слой:** Сервер + Раннер.

**Что нужно:**
- 3 уровня тревоги Локации (синий/жёлтый/красный) с опциональной кастомизацией админом (число и палитра).
- Изменение уровня — у СОДЦ.
- Уведомления всем ответственным операторам при повышении.
- **Кнопка «Перейти к Локации»** в уведомлении (только для роли СОДЦ): по клику переходит на вкладку «Локации» с предвыбранной локацией и развёрнутым списком её датчиков, чтобы быстро отключить ложно срабатывающие.
- Включение/отключение датчиков (точечно + массово через фильтр).

**Эксклюзив:** **автоматическое повышение уровня тревоги** при определённых паттернах инцидентов (через JSONLogic-правила); **аудит изменений уровня с обоснованием** обязательно (causal comment).

**Acceptance:** при возникновении пожарной тревоги СОДЦ повышает уровень Локации до красного — все операторы видят соответствующее уведомление.

### P1.5. Дашборд статистики

**Источник:** SecurOS §11.
**Слой:** Сервер + Раннер.

**Что нужно (минимум — повторить SecurOS):**
- 4 диаграммы: Общее по статусам, По Локациям, По типам, По операторам ПУ + ср. время реакции.
- Фильтры (наследуются от Архива).
- Переключение Гистограмма/Кластеры.

**Эксклюзив (наш):**
- **Экспорт данных** в CSV/Excel/Parquet.

**Слой:** Сервер.

**Что нужно:**

**Эксклюзив:** **GraphQL** опционально для гибких запросов; **готовые SDK** на TypeScript / Python / Go / Java / Kotlin.

**Acceptance:** клиент пишет свой Python-скрипт, который слушает инциденты, делает свои отчёты, шлёт в Telegram-бот — всё через API; крупный клиент включает outbound в свой корпоративный Data Center и получает все события надёжно с retry.

### P2.11. VMS-агностичный adapter-слой источников событий (главный стратегический эксклюзив)

**Источник:** обратное от слабости SecurOS Dispatcher (см. `competitor-securos-dispatcher.md` §17) и IM ITV (привязан к Intellect).
**Слой:** Go-бэкенд (адаптеры) + Сервер (нормализованная модель источников) + Конфигуратор (UI выбора и настройки адаптера).

- **SecurOS Dispatcher** жёстко вшит в SecurOS VMS: датчики, типы датчиков, типы событий, камеры, пользователи, видео, карта, БД — всё через объекты SecurOS. Не работает поверх ничего другого.

- Карта с автодетектом: GIS-координаты → Leaflet/Mapbox; растровые карты адаптера → как fallback.

#### Эксклюзив сверху
- **Гетерогенный режим:** один и тот же инцидент может приходить с источников разных VMS одновременно (например, инцидент «пожар в здании А» — событие с пожарного датчика по MQTT + видео с Milestone-камеры + СКУД-событие через REST).
- **Hot-swap адаптера:** замена VMS под существующим инвентарём без потери истории инцидентов.
- **Проксирование макросов:** action `callMacro` в DSL → адаптер транслирует в нативный механизм VMS (Intellect macros, SecurOS scripts, Milestone Rules, и т.д.).
- **Открытый SDK для адаптеров:** партнёры могут написать свой адаптер на Go и опубликовать.

#### Acceptance
- На демо: подключаем три разные VMS (Axxon Next, SecurOS, Milestone) к одному инстансу нашего сервера; в одной ленте инцидентов видны события со всех трёх; в карточке инцидента работает видео и карта от соответствующей VMS.
- В полевом тесте: клиент с Milestone-инфраструктурой устанавливает наш продукт и за 1 час подключает все свои камеры — без замены VMS.

#### Маркетинговая позиция
**«Менеджер инцидентов AxxonSoft работает поверх любой VMS — не заставляет менять то, что у вас уже есть».** Эта формулировка **закрывает SecurOS Dispatcher на тендерах с гетерогенной инфраструктурой** и параллельно открывает нам тендеры с конкурирующими VMS, куда SecurOS Dispatcher даже не проходит на этап оценки.

## Приложение: чек-лист для PRD каждой фичи <a id="prd-checklist"></a>

При проработке любой фичи из этого документа использовать единый чек-лист, чтобы не упустить интеграцию между слоями:

1. **Изменения в DSL** — есть/нет, какие новые поля/типы, обратная совместимость, миграция старых сценариев?
2. **Изменения в JSON Schema** — обновления для уровня 1 валидации.
3. **Изменения в семантической валидации** (уровень 2 на сервере).
4. **Изменения в БД** (миграция, индексы, ограничения).
5. **Изменения в OpenAPI** (новые эндпоинты, deprecated).
6. **Изменения в `configurator-web`** (новые экраны, компоненты, диагностики).
7. **Изменения в `runner-web`** (новые компоненты, состояние, переходы).
8. **Изменения в нативных runner’ах** (iOS/Android — что должны рендерить).
9. **Backwards compatibility** — что будет со старыми инцидентами / сценариями / клиентами.
10. **Документация** — обновить `dsl-v1-draft.md` (если затрагивает DSL), README, CHANGELOG.
11. **Тесты** — unit, integration, e2e, regression на `examples/`.
12. **Локализация** — все строки в i18n файлы.
13. **Доступность** — ARIA, клавиатура, контраст.
14. **Безопасность** — авторизация, валидация ввода, защита от XSS/SQLi.
15. **Перфоманс** — нагрузочное тестирование при N инцидентов / M операторов.
16. **Telemetry** — логи, метрики, трассировки.
17. **Аудит** — пишется ли изменение в журнал аудита.
18. **Multi-tenancy** — изоляция данных тенантов.
19. **Demo-сценарий** — для маркетинга и продаж.

## Завершение

Этот документ — **живой**, должен пересматриваться по мере выхода новых версий конкурентов и появления обратной связи от клиентов. Следующий пересмотр — **через 3 месяца** или при существенном анонсе SecurOS / запуске IM-замены конкурентов.

**Главный посыл:** у нас есть **уникальная архитектурная база** (Server-Driven UI + JSONLogic + версионирование) **плюс возможность стать единственным VMS-агностичным менеджером инцидентов на рынке**. Если мы добавим **операционную зрелость SecurOS** — получим лучший в классе продукт. Если зацикливаемся только на технологиях без операционной обвязки — останемся «классно спроектированной игрушкой» против рабочей лошадки SecurOS.

**Стратегия проста:** каждые 3 месяца закрываем один блок из P0/P1, параллельно закладываем P2-эксклюзивы (особенно **P2.11 — VMS-агностичный adapter-слой как абсолютный приоритет**), и через год можем демонстрировать **превосходство по всем 23 категориям** из таблицы оценки в `competitor-comparison.md`, плюс уникальную позицию «работает поверх любой VMS».
